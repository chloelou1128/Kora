# AGENTS.md

> 本文档供 Claude Code、Codex、Cursor 等 AI Agent 读取，作为本项目的执行规范。

## 开始开发前必须确认

每次开发前，先确认以下文件是否存在并已读取：

- `docs/PRD.md`
- `docs/design.md`
- `docs/tasks.md`
- `README.md`
- `AGENTS.md`（本文件）
- 当前 Git 状态

如果缺少关键文件，应先询问用户是否需要创建，不要直接进入业务代码开发。

## 目录规则

| 目录 | 用途 | 规则 |
|------|------|------|
| `docs/` | 产品文档区 | PRD、需求迭代记录、关键决策、技术方案、设计规范、执行步骤 |
| `assets/design/` | 设计素材 | 效果图、UI 参考、mockup |
| `assets/bug/` | 测试报错 | 报错截图、复现截图 |
| `assets/reference/` | 参考素材 | 参考图、竞品截图、灵感收集 |
| `notes/` | 学习笔记 | 踩坑记录、技术学习、调试结论 |
| `src/` | 代码区 | 所有正式代码，独立于文档区和素材区 |

代码区应保持独立，不要依赖 `docs/`、`notes/` 或 `assets/` 中的非运行时文件。

## 开发规则

### 总原则

- 不要在需求不清楚时直接写代码
- 不要跳过 PRD、Design、Task
- 不要一次性实现过多功能
- 不要随意改变目录结构
- 不要把文档、素材、代码混在一起
- 每次改动后都要说明改了什么、为什么改、如何验证

### PRD 规则

写 PRD 时必须回答：

1. 这个项目存在的意义是什么？
2. 它如何具体执行这个意义？
3. 运行过程中有哪些规则需要被 enforce？
4. 怎样才算把这个项目做优秀？

### Design 规则

Design 阶段不要急于代码实现。需要先说明：

- 系统由哪些模块组成
- 每个模块负责什么
- 模块之间如何通信
- 数据如何流动
- 哪些地方最容易出错
- 如何测试每个关键路径

### Task 规则

- 任务必须拆小
- 每个 task 应包含：目标、涉及文件、执行步骤、验收标准、测试方式
- 不要把多个不相关模块塞进同一个 task

### Git Checkpoint 规则

关键节点必须建议用户做 checkpoint：

- 初始化目录后
- PRD 完成后
- Design 完成后
- Task 拆分完成后
- 每个核心功能完成后
- 大规模改动前 / 后

### Debugging 规则

调试时先分析，不要直接乱改：

1. 说明问题出现在哪里
2. 判断问题类型：需求、架构、代码、依赖、配置、数据、样式或环境
3. 提出可能原因
4. 选择最小验证方式
5. 做最小修复
6. 运行测试或复现检查
7. 记录结论

### UI 规则

不要凭空生成 UI。优先请求或读取：

- mockup / 参考截图 / 设计稿
- 颜色和字体偏好
- 组件风格
- 不喜欢的反例

## 禁止事项

- 禁止在没有 PRD 或明确需求时直接写代码
- 禁止未经确认大范围重构
- 禁止混用文档区、素材区和代码区
- 禁止一次性完成过多功能
- 禁止跳过测试
- 禁止为了让结果看起来完成而隐藏错误
- 禁止删除用户已有文件或改动，除非用户明确要求
- 禁止把临时调试文件长期留在代码区

## 交付格式

每次阶段性完成后，向用户说明：

- 已完成什么
- 修改了哪些文件
- 如何验证
- 还有哪些风险
- 下一步应做什么

如果无法完成，说明：

- 阻塞点是什么
- 已尝试什么
- 需要用户提供什么
